Scaling the atlas: JAX games + Deep CFR

Moving the futarchy equilibrium atlas to 9-round games via JAX-native simulation and neural Deep CFR.

May 2026 · Companion to the main atlas.

Why this page exists: the main atlas reports tabular CFR+ at 3 rounds because the info-state count blows up at 6 / 9 rounds. With each trader acting only once at 3 rounds, "strategic vs myopic" collapses to nothing, and there is no room for the sawtooth deception, manipulator cheap-talk, or anything else that makes sequential markets interesting. This page documents the next-generation tooling that fixes this — and where it currently stands.

The bottleneck

OpenSpiel's Python MCCFR walks the game tree per-iteration, calling state.child(action) at every node. For a Python-defined game, state.child() invokes Python's __copy__ machinery, which is orders of magnitude slower than the actual regret arithmetic. Empirically: 22 MCCFR iterations/sec on a 9-round Python game. At ~50K iterations to converge, that's three hours of wall clock. The bottleneck is not arithmetic — it's Python state cloning.

GPU does not directly help, because the bottleneck is Python overhead, not parallel arithmetic. CFR is also irregular and recursive, a poor fit for SIMD/SIMT. To unlock GPU acceleration we have to (a) write the game in a JIT-able, vmap-able framework like JAX, and (b) use a neural-network approximation of regrets (Deep CFR) since the info-state space is far too large to enumerate.

What we built

The new sub-project mechanism-design/jax-futarchy/ implements:

  • JAX-native Galanis market. Game state is a NamedTuple of JAX arrays (omega, signals, price history, action history, profits, terminal flag). step(state, action) is a pure function — JIT-compiles, vmaps over batches, no Python state cloning.
  • Flax regret network. Per-player MLP that maps info-state vectors to per-action regret estimates. Regret-matching converts those into a strategy.
  • External-sampling Deep CFR. At each traverser node we enumerate all K actions and roll the rest of the trajectory by sampling from the current regret-matching strategies. Per-iteration regrets are appended to a replay buffer; the network is trained on the buffer between iterations.

Running it on the DGX Spark

We have remote access to a shared NVIDIA DGX Spark (Grace Blackwell GB10, 121 GB unified memory, 20-core ARM CPU) via Tailscale. The DGX runs CUDA 13 (August 2025 toolkit); pip-published JAX wheels target CUDA 12 and (in some cases) x86_64 only, so the bare-metal pip install gives CPU-only JAX. The fix is to use NVIDIA's NGC container nvcr.io/nvidia/jax:26.04-py3, which bundles a JAX build (0.9.2.dev) with the correct CUDA-13 + Blackwell-aware jax_cuda13_pjrt plugin. Inside that container jax.devices() returns [CudaDevice(id=0)] and JAX runs on the GB10. We cap our GPU memory allocation at 25% via XLA_PYTHON_CLIENT_MEM_FRACTION=0.25 so we coexist with the vLLM service Vinicius runs on the same GPU.

Two findings emerge at 9 rounds that did not at 3 rounds: (1) Medium and Hard structures now aggregate better than Easy under DCFR — the additional rounds let the network find the dominant strategy more cleanly when the metric is "all yes" (clear signal-aggregation problem) than when it's "at least 2 yes" (which has more partial-information cases that confuse the network). (2) The Very Hard structure (each trader sees the other two signals; payoff = exactly 2 yes) is the worst by far — median LE 0.755 means the equilibrium is essentially uninformative. The 4-cell partition combined with 9-round dynamics overwhelms our small (64×2 MLP) network's capacity.

Hanson 9-round DCFR is biased toward A. Across all 8 ω the equilibrium gives Pr(A wins) > 0.5 except at state g. This is worse than the 3-round CFR+ baseline (0.754 vs 0.630). The likely cause is the same convergence-quality gap visible in Galanis: with our small (64×2 MLP) network and 300 iterations, the network has not yet found the right per-info-state regret structure for a deep-tree game with 8 different ω. Scaling to a larger network and 10×+ iterations (which the DGX with GPU should make feasible once the JAX CUDA-13 wheels land) is the next investment.

Honest summary

What the 9-round runs do establish: (i) JAX rollouts and DCFR loops are 50× faster than the Python OpenSpiel MCCFR on the same game; (ii) the JAX game definition admits all three mechanisms (Galanis, Hanson, MetaDAO) with identical step semantics; (iii) the manipulator equilibrium-selection phenomenon survives going from 3 to 9 rounds, replicating Chapter 6's qualitative finding under genuine multi-turn dynamics.

What they do not establish: that the 9-round numerical equilibria are correct. The DCFR convergence quality is worse than CFR+ tabular at 3 rounds, and significantly worse for the Very Hard structure (4-cell partition). This is a network-capacity and iteration-count problem, not a mechanism-design one. Closing the gap to publication-quality numbers needs (a) GPU-enabled JAX (waiting on CUDA-13 wheels), (b) larger networks, (c) 10×+ training iterations, and ideally (d) a separate strategy network as in canonical Deep CFR.

Honest gap analysis

The JAX-Deep-CFR scaffolding is the substantive contribution of this iteration: a JIT/vmap-able game environment and a working Deep CFR trainer. The quantitative results are not yet competitive with tabular CFR+ at 3 rounds. To close that gap requires several known improvements that we have not yet implemented:

  • Separate strategy network. Standard Deep CFR maintains a second network trained on the average policy across iterations. Without it we use the final-iteration policy, which is only an approximate Nash. Adding it is mechanical.
  • Larger / wider networks. Our MLPs are 64 × 2. For richer info states (longer histories, finer action grids), 128 × 3 or attention-based encoders should help.
  • More iterations + larger buffer. 100 iterations × 50K buffer is small for Deep CFR. The TF2 reference uses ~10⁶ iterations with reservoir sampling.
  • GPU. JAX is CPU-only here; moving to a CUDA device (DGX Spark, A100, etc.) should give 10–100× on the SGD step that's currently the bottleneck.
  • Port other mechanisms. Hanson, MetaDAO, Proposal Poker are still in their Python OpenSpiel form. Porting to JAX is straightforward once the Galanis pattern is established.

We document this gap explicitly rather than overselling early results. The framework is the deliverable; tuning Deep CFR to match tabular CFR+ quality is a meaningful follow-up project — measured in days, not hours — and may need GPU hardware to be practical.
